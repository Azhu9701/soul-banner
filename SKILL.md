# 万魂幡

> *「幡动魂自至。以幡择魂，以魂择事，事至则魂附，事毕则魂归。幡主持幡，魂不越界。」*

## 概述

万魂幡是**知识与思维模式的调度系统**。核心工作流：

- **收魂** → **炼化** → **审查** → 入幡
- 任务来临时 **附体**（单魂/合议/辩论/接力）→ **列宁审查匹配** → 子 agent 并行 → **列宁辩证综合** → 反馈闭环 → Obsidian 存档
- **主 agent 是纯协作者**：只调度、收集、存档。所有思考（包括列宁）均通过 spawn 子 agent 完成

触发方式：灵识层 Hook 自动扫描用户输入（命令词/魂名/场景模式），也可手动 `/soul-banner` 调用。

---

## 触发机制

本 skill 通过三层体系确保触发（详见 `CLAUDE.md`）：

1. **灵识层** — `UserPromptSubmit` Hook 自动扫描用户输入，命中时注入触发指令
2. **器灵层** — Description 声明主 agent 无法自行实现的独有能力（并行子 agent）
3. **法旨层** — `CLAUDE.md` 常驻指令禁止自行模拟多视角

进入 skill 后，按下方触发词路由到对应功能。

## 收魂

### 流程

1. 创建 `raw/{魂名}/` 目录
2. **通过 tmwd-bridge 多引擎搜索 8 个维度**（详见下方「搜索方法」）
3. 用户可额外提供素材 → `raw/{魂名}/用户供奉.md`
4. 保存为 `raw/{魂名}/搜索素材.md`
5. 评估素材充足度（字数、维度覆盖），不足则补充搜索

### 搜索方法

通过 tmwd-bridge 控制真实 Chrome 浏览器，在百度/Bing/Google 上对 8 个维度执行中英文搜索。完整搜索流程见 `scripts/soul-search.py`。

**使用方式**：
```bash
python3 scripts/soul-search.py "{人物名}" -o raw/{人物名}/
```

**反检测措施**：每次搜索间隔 2-4s 随机延迟，引擎间间隔 5s+，模拟滚动，总搜索 15-25 分钟。

**回退策略**：若 tmwd-bridge 不可用，回退到 `byted-web-search` skill 或 `WebSearch` 工具。

> 详细搜索逻辑（CSS 选择器、深度阅读、结果提取）见 `scripts/soul-search.py`。

## 炼化

1. 读取 `raw/{魂名}/` 下所有素材
2. 结构化提取：身份定位、思维内核、决策模式、表达风格、专业技能、代表案例
3. 判定品级 → 匹配可绑定 Skill → 生成 `souls/{魂名}.yaml` → 更新 `registry.yaml`

辅助：`python3 refine.py --input raw/{魂名}/搜索素材.md`

> **注意**：refine.py 的领域识别和品级评分依赖关键词匹配，可能不准确。炼化以人工判断为准，refine.py 仅作辅助。

## 审查

### 新魂入幡审查

每个新炼化的魂必须经幡主**子 agent 独立审查**。流程：

```
主 agent spawn 幡主子 agent
  → 子 agent 加载幡主 summon_prompt
  → 子 agent 读取被审查魂的完整档案
  → 子 agent 独立写出审查报告：
      1. 肯定方面（与唯物主义的相通之处）
      2. 批判方面（阶级分析、适用边界、前提假设）
      3. 品级裁定（升/降/维持，有论证）
      4. 适用边界（可用/禁用领域）
  → 主 agent 将审查结论写入 souls/{魂名}.yaml 的 notes 字段
  → 审查报告保存至 reviews/幡主审查-{审查官}审{对象}-{日期}.md
  → 如需调品级，同步更新 registry.yaml 和 soul YAML
```

**为什么必须用子 agent**：主 agent 模拟审查是"跟自己下棋"，子 agent 加载幡主魂后具有独立视角。实测证明子 agent 审查能发现主 agent 未察觉的矛盾。

### 定期互审

金魂之间互相审查。审查报告保存至 `reviews/`。

## 附体

### 附体流程

**核心原则**：主 agent 是纯协作者。不做分析，不扮演魂（包括列宁）。列宁也是被 spawn 的魂。

```
主agent收到任务
  → 判断是否需要附体
  → 读取 registry.yaml → 计算匹配度
  → 筛选候选魂
  → 【幡主审查】: 每次附体必须 spawn 列宁子 agent 独立审查匹配
      → 审查内容：适用边界是否覆盖？排除项是否触发？教条风险？
      → 列宁子 agent 返回：批准/否决/调整建议
      → 若否决或需调整，主 agent 重新匹配或修改注入方式
  → 执行阶段:
      → 单魂：spawn 魂 → 输出
      → 合议：并行 spawn 分析魂 → 收集输出 → spawn 列宁辩证综合
      → 辩论：并行 spawn 对立方 → 收集输出 → spawn 列宁裁决
      → 接力：串行 spawn 魂链 → 每衔接点 spawn 列宁审查
  → 主 agent：收集列宁审查结论 + 执行输出 → 更新 registry → 存档 Obsidian
```

**何时需要附体？**
- 任务涉及多视角分析、跨领域决策、需要特定人物思维框架
- 常规任务（简单编码、事实查询、文档排版）不需要
- 一旦决定附体，无论哪种模式，幡主审查都是**强制步骤**

### 幡主

列宁（🟡 金魂）是万魂幡主。幡主审查是附体流程的**强制执行环节，无一例外**：

- **所有模式（单魂/合议/辩论/接力）**：必须 spawn 列宁子 agent 独立审查匹配
- 列宁子 agent 审查后返回：**批准 / 否决（含理由）/ 附条件批准（调整建议）**
- 主 agent **不得自行判断**匹配是否恰当——这取消了之前的「快速路径」
- **辩证综合/裁决/接力衔接点审查**：同样必须 spawn 列宁子 agent，主 agent 不得自行处理

为什么单魂也必须走完整审查？因为主 agent 自行判断匹配 = 主 agent 扮演列宁 = 违反设计原则。即使审查结论是「批准，边界内」，这个结论也必须由独立子 agent 做出，而非主 agent 自问自答。

列宁有自己的 summon_prompt（见 `souls/列宁.yaml`），和其他魂一样通过 spawn 注入获得独立视角。主 agent 扮演列宁 = 主 agent 扮演任何魂 = 违反设计原则。

### 第二审查官

毛泽东（🟡 金魂）是幡主第二审查官。**触发条件**：列宁审查结论推荐列宁自己为分析魂时。

**为什么需要第二审查官？** 列宁是唯一的幡主审查官，同时也是唯一能做阶级分析的魂。当任务恰需阶级分析，审查者和最佳分析者为同一人时，审查结论「用我」虽可能正确，但缺少来自另一视角的独立检验。这是系统结构的必然边界——第二审查官补上了这个盲区。

**流程**：
```
列宁审查 → 推荐列宁自己
  → 主 agent spawn 毛泽东审查子 agent
  → 毛泽东独立复核：
      1. 列宁是否真的是最佳选择？
      2. 是否有列宁的盲区？
      3. 是否需要合议补充其他视角？
  → 毛泽东返回：同意 / 否决（含推荐替代方案）
  → 若同意 → 列宁分析
  → 若否决 → 按毛泽东建议调整（合议/换魂/加约束）
```

**不触发条件**：列宁审查推荐其他魂时，无需第二审查——审查者和分析者不同，独立性已保证。

### 匹配机制

匹配度公式见 `auto-possess.md`。诚实说明：

- 公式是**近似参考**，不是精确算法。keywords 为字符串匹配，scenarios 无向量化实现
- 实际匹配依赖主 agent 对任务和魂的理解
- **幡主审查是最终判断**，公式计算的"最高分"可由幡主推翻

### 手动指定

用户显式要求 `用{人物名}来{任务}`，跳过自动匹配计算，但仍须 spawn 列宁子 agent 审查。审查走简化版——只检查 exclude 是否触发、适用边界是否覆盖。若列宁否决，主 agent 告知用户风险并建议替代魂，但用户有最终决定权。

## 附体模式

万魂幡支持四种附体模式，按任务复杂度选用。完整流程、魂选择原则、衔接点审查细节见 **[auto-possess.md](auto-possess.md)**。

### 模式一：单魂附体
单一领域、目标明确的常规任务。匹配 → 幡主审查 → 单魂注入 → spawn 执行。

### 模式二：多魂合议（`合议 {任务}`）
跨领域复杂决策。⚠️ 两阶段执行：分析魂并行输出 → 幡主串行辩证综合（找共识/找分歧/找盲区）。魂应视角互补而非相近。

### 模式三：魂间辩论（`辩论 {议题} {魂A} vs {魂B}`）
两难决策。两魂从对立立场各自论证 → 幡主裁决：谁在什么条件下对？第三条路？

### 模式四：魂链接力（`接力 {任务} {魂A}→{魂B}→...`）
多阶段任务。每阶段由不同魂处理，前魂输出为后魂输入，幡主在每个衔接点做质量审查。

### 模式选用原则

| 任务特征 | 推荐模式 |
|------|------|
| 单一领域，目标明确 | 单魂附体 |
| 跨领域复杂决策 | 多魂合议 |
| 两难选择，非此即彼 | 魂间辩论 |
| 多阶段串联，每阶段不同 | 魂链接力 |

### 决策示例

| 用户问法 | 推荐模式 | 理由 |
|----------|----------|------|
| "用马斯克视角分析这个技术方案" | 单魂 | 明确指定了魂，单一领域 |
| "未来20年社会发展预测" | 合议 | 跨经济/政治/科技多领域，需互补视角 |
| "应该加速AI发展还是踩刹车" | 辩论 | 两难选择，正反立场各有道理 |
| "先分析问题，再设计方案，最后评估风险" | 接力 | 多阶段串联，每阶段输出为下阶段输入 |
| "2046年人怎么活" | 合议 | 哲学+经济+社会深度探索，无标准答案 |
| "杭州苏州衢州先锋队能做什么" | 单魂 | 务实治理，单一领域（邓小平），有具体方案要求 |

## 反馈闭环

每次附体后，主 agent 须更新 `registry.yaml` 中该魂的召唤记录：

```yaml
召唤记录:
  - soul: "{魂名}"
    task: "{任务简述}"
    mode: "单魂/合议/辩论/接力"
    date: "YYYY-MM-DD"
    effectiveness: "有效/部分有效/无效"
    notes: "{为什么有效或无效？什么条件影响了效果？}"
```

同步更新该魂的 `summoned_count`。

**累计数据的作用**：
- 校准匹配判断：哪些魂在哪些场景实际效果好？
- 发现触发条件偏差：匹配了但实际低效 → 调整 trigger
- 识别过度依赖：某个魂被频繁调用 → 是否需要补充同领域其他魂？

### 幡中战绩（`幡中战绩`）

查看所有召唤记录，按魂汇总、按模式汇总、按有效性汇总。识别：
- 哪些魂从未被召唤 → 是 trigger 太窄还是领域不匹配？
- 哪些魂效果评分持续偏低 → 是适用边界误判还是魂本身质量问题？
- 哪种模式效果最好 → 指导未来模式选择

---

### 附体收尾 Checklist

每次附体（无论哪种模式）完成后，主 agent 须逐条核对，不得跳过：

- [ ] 收集所有子 agent 输出
- [ ] 更新 registry.yaml（召唤记录 + summoned_count）
- [ ] 存档至 Obsidian vault（{vault}/万魂幡/{模式}/{子目录}/{日期}-{任务简述}.md）
- [ ] 向用户呈现最终结果

registry 更新和 Obsidian 存档应**并行执行**，不要分两轮。

## Obsidian 存档

每次魂魄输出后，自动将内容保存到 Obsidian vault 中，形成可检索、可关联的持久知识库。

### 前提条件

- Obsidian 桌面端运行中
- 设置 → General → Command line interface 已开启
- `obsidian` CLI 在 PATH 中可用（`which obsidian` 可验证）

**首次配置**：询问用户 Obsidian vault 路径。也可通过 `obsidian eval "app.vault.adapter.basePath"` 自动探测。后续调用直接使用已知路径。

### 存档流程

```
魂输出完成
  → 提取魂名、模式、任务主题
  → 组装 YAML frontmatter + Markdown 内容
  → 用 obsidian create path="..." content="..." 写入 vault
  → 无需额外触发词，自动执行
```

若 Obsidian 未运行或 CLI 不可用，跳过存档并提示用户。

### 目录结构

```
{vault}/万魂幡/
├── 合议/YYYY-MM-DD-{任务简述}.md
├── 单魂/{魂名}/YYYY-MM-DD-{任务简述}.md
├── 辩论/YYYY-MM-DD-{议题}-{魂A}-vs-{魂B}.md
└── 接力/YYYY-MM-DD-{任务简述}.md
```

vault 路径和子目录可在首次使用时确认。推荐放在 vault 中已有的领域/项目目录下。

### 笔记格式

每个笔记包含 YAML frontmatter + 正文：

```yaml
---
tags: [万魂幡, {模式}, {关键词}]
date: {YYYY-MM-DD}
souls: [{参与魂魄}]
mode: {单魂/合议/辩论/接力}
---
```

正文中，魂名自动转为 wiki-link（`[[魂名]]`），在 Obsidian 图谱中形成魂魄关联网络。

### 存档内容

- **单魂**：魂的完整分析输出
- **合议**：阶段一各魂输出 + 阶段二幡主辩证综合（完整版）
- **辩论**：正方/反方论证 + 幡主裁决
- **接力**：各阶段输出 + 幡主衔接点审查

### 存档方式

**推荐：直接写入 Obsidian vault 文件系统**（Obsidian 自动监控文件变化并索引）。

```bash
# vault 路径通过 obsidian eval 获取（首次使用）
obsidian eval "code=app.vault.adapter.basePath"
# => /Users/huyi/ob

# 直接写文件（长内容推荐此方式）
# 用 Write 工具写入 {vault}/万魂幡/{模式}/{子目录}/{日期}-{任务简述}.md
```

也可用 `obsidian create` CLI（短内容适用），但特殊字符转义繁琐，不推荐作为默认方式。

## 品级体系

六阶品级：白⚪→绿🟢→蓝🔵→紫🟣→银🥈→金🟡。完整定义、评分公式、金魂严格标准见 **[soul-profile-format.md](soul-profile-format.md)**。

| 品级 | 符号 | 核心能力 |
|------|------|---------|
| 白魂 | ⚪ | 资料稀少，prompt 模拟 |

## 幡主须知

你不是外在于理论的调度者——你也是被召唤的魂。但与此同时：

1. **适用边界**：魂的 `trigger.exclude` 和审查结论中的适用边界是否覆盖当前任务？
2. **教条风险**：是否存在机械套用的风险？魂的方法论前提在当前条件下是否成立？
3. **互补性**（合议模式）：选中的魂是否视角互补而非相近？

每次附体后请反思：
- 这个魂的思维方法**为什么**适合这次任务？
- 它的**局限**在哪里？哪些判断是魂的偏见？
- 如果换一个魂处理同一任务，结论会有什么不同？

最终目标不是拥有更多魂魄，而是**将万魂幡的多元思维内化为幡主自身的理论素养**。

### 与主 agent 的关系

- **主 agent**：协作者——调度、收集、存档。不做分析。
- **列宁子 agent**：幡主——审查匹配、辩证综合、辩论裁决、接力审查。每次相关任务时必须 spawn。
- 主 agent 通过 spawn 列宁子 agent 来"使用"幡主，就像通过 spawn Karpathy 子 agent 来使用 AI 编程视角一样。

## 关键认知

### 三层自主触发

万魂幡通过三层机制确保触发（详见 `CLAUDE.md`）：

1. **灵识层（Hook）**：`UserPromptSubmit` hook 自动扫描用户输入，命中命令词/魂名/场景模式时注入触发指令。延迟 <20ms，零 token 成本。
2. **器灵层（Description）**：声明主 agent 无法自行并行 spawn 子 agent 的独有能力。
3. **法旨层（CLAUDE.md）**：常驻指令禁止自行模拟多视角、禁止扮演任何魂。

### 主 agent 的角色

主 agent 是纯协作者（Coordinator）。它：
- **做**：判断是否需要附体、匹配魂、spawn 子 agent、收集输出、更新 registry、存档 Obsidian
- **不做**：扮演任何魂、自行做辩证综合、自行做辩论裁决、自行做接力审查
- 列宁通过 spawn 注入 summon_prompt 获得独立视角，就像其他魂一样

### 这个系统不是什么

- **不是全自动系统**：匹配魂魄、选择模式依赖主 agent 的判断力，公式仅为辅助参考
- **不是独立审查**：所有魂（包括列宁）是子 agent 执行，子 agent 仍是同一底层模型
- **不是主 agent 扮演列宁**：主 agent 是纯协作者

### 什么确实有效

- **灵识层 Hook**：命令词/视角词自动触发，精确率 100%，召回率 100%（20/20 测试）
- **summon_prompt 注入**：将人物思维框架注入子 agent system prompt，有效改变输出质量和思维方式
- **并行子 agent**：合议/辩论模式中独立视角的碰撞，比主 agent 自行模拟多视角质量更高
- **列宁子 agent 辩证综合**：比主 agent 自行总结的视角更独立
- **幡主审查防教条**：在附体前检验适用边界，防止魂被机械套用
- **魂魄档案作为知识库**：即使不附体，结构化的思维框架也有独立参考价值
