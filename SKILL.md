# 万魂幡

> *「幡动魂自至。以幡择魂，以魂择事，事至则魂附，事毕则魂归。幡主持幡，魂不越界。」*

## 概述

万魂幡是**知识与思维模式的调度系统**：

- **收魂**：联网搜索 + 用户供奉，收集目标人物的多维度素材
- **炼化**：将原始素材结构化为 Soul Profile YAML，并判定品级
- **附体**：spawn 子 agent 时匹配魂魄注入 summon_prompt，改变子 agent 的思维模式
- **幡主审查**：幡主在附体前审查匹配是否恰当，防止教条式套用
- **独立审查**：新魂入幡须经幡主子 agent 独立审查，非主 agent 模拟
- **管理**：查看品级、升级魂魄、散魂

---

## 触发词

| 触发词 | 功能 |
|--------|------|
| `收魂 {人物名}` | 联网搜索 8 个维度 + 可选用户供奉 |
| `炼化 {人物名}` | 将 raw 素材炼化为 Soul Profile，判定品级 |
| `幡中有什么魂` | 查看已炼化魂魄列表及品级分布 |
| `散魂 {人物名}` | 删除魂魄档案（保留 raw 素材） |
| `升级魂魄 {人物名}` | 重新收魂 → 炼化 → 幡主审查 → 覆盖原档案 |
| `审查 {人物名}` | 幡主子 agent 独立审查指定魂魄 |
| `用{人物名}来{任务}` | 手动指定魂魄执行任务（跳过自动匹配） |

## 收魂

1. 创建 `raw/{魂名}/` 目录
2. 联网搜索 8 个维度：核心思维框架、决策模式、管理风格、技术愿景、沟通风格、工作方法、代表性言论、争议素材
3. 用户可额外提供素材 → `raw/{魂名}/用户供奉.md`
4. 保存为 `raw/{魂名}/搜索素材.md`
5. 评估素材充足度，不足则补充搜索

---

## 炼化

1. 读取 `raw/{魂名}/` 下所有素材
2. 结构化提取：身份定位、思维内核、决策模式、表达风格、专业技能、代表案例
3. 判定品级 → 匹配可绑定 Skill → 生成 `souls/{魂名}.yaml` → 更新 `registry.yaml`

辅助：`python3 refine.py --input raw/{魂名}/搜索素材.md`

> **注意**：refine.py 的领域识别和品级评分依赖关键词匹配，可能不准确。炼化以人工判断为准，refine.py 仅作辅助。

## 审查

### 新魂入幡审查

每个新炼化的魂必须经幡主**子 agent 独立审查**。流程：

```
主 agent spawn 幡主子 agent
  → 子 agent 加载幡主 summon_prompt
  → 子 agent 读取被审查魂的完整档案
  → 子 agent 独立写出审查报告：
      1. 肯定方面（与唯物主义的相通之处）
      2. 批判方面（阶级分析、适用边界、前提假设）
      3. 品级裁定（升/降/维持，有论证）
      4. 适用边界（可用/禁用领域）
  → 主 agent 将审查结论写入 souls/{魂名}.yaml 的 notes 字段
  → 审查报告保存至 reviews/幡主审查-{审查官}审{对象}-{日期}.md
  → 如需调品级，同步更新 registry.yaml 和 soul YAML
```

**为什么必须用子 agent**：主 agent 模拟审查是"跟自己下棋"，子 agent 加载幡主魂后具有独立视角。实测证明子 agent 审查能发现主 agent 未察觉的矛盾。

### 定期互审

金魂之间互相审查。审查报告保存至 `reviews/`。

## 附体

### 附体流程

```
主agent收到任务
  → 判断需要 spawn 子 agent
  → 读取 registry.yaml
  → 遍历每个魂的 trigger，计算匹配度
  → 筛选匹配度 ≥ 70 的候选魂
  → 【幡主审查】判断匹配是否恰当
      → 候选魂的适用边界是否覆盖当前任务？
      → 是否存在教条式套用的风险？
      → 是否需要组合多个魂？
  → 幡主批准 → 注入 summon_prompt → spawn 执行
  → 幡主否决 → 不附体或换魂
  → 无候选魂 → 不附体
```

### 幡主

幡主在 `registry.yaml` 中指定。幡主审查是附体流程的**强制执行环节**：

- **重要任务**：spawn 幡主子 agent 独立审查匹配（推荐）
- **常规任务**：主 agent 以幡主方法论直接判断（快速路径）

### 匹配机制

匹配度公式见 `auto-possess.md`。诚实说明：

- 公式是**近似参考**，不是精确算法。keywords 为字符串匹配，scenarios 无向量化实现
- 实际匹配依赖主 agent 对任务和魂的理解
- **幡主审查是最终判断**，公式计算的"最高分"可由幡主推翻

### 手动指定

用户显式要求 `用{人物名}来{任务}`，跳过自动匹配和幡主审查，直接注入。但主 agent 应提醒如果魂的适用边界不覆盖当前任务。

## 品级体系

### 六阶品级

| 品级 | 符号 | 分数 | 定义 | 能力 |
|------|------|------|------|------|
| 白魂 | ⚪ | 0-30 | 资料稀少，仅公开信息 | prompt 模拟 |
| 绿魂 | 🟢 | 31-60 | 资料中等 | prompt + 1-2 Skill |
| 蓝魂 | 🔵 | 61-90 | 资料充足 | prompt + Skill链 + 执行策略 |
| 紫魂 | 🟣 | 91-115 | 资料丰富，有明确方法论 | + 方法论模板，但依赖天赋不可复制 |
| 银魂 | 🥈 | 116-150 | 领域顶级实践者，方法论完整可操作 | + 系统化方法论 + 概念创造力，**但缺独立世界观和方向判断** |
| 金魂 | 🟡 | 151+ | 独立世界观，有方向判断力 | + 知识库 + 持久记忆 + 全技能链 |

### 金魂严格标准

金魂必须同时满足：
1. **执行方法**：有完整的、可操作的方法论体系
2. **判断方法**：有独立世界观，能在历史变局中判断方向——知道往哪走、为谁服务
3. **自我批判**：体系内有自我修正机制

**缺第2条但有第1条 → 银魂。有第2条但方法论不可复制 → 紫魂。**

品级评分公式见 `soul-profile-format.md`。公式是初始参考，最终品级由炼化判断 + 幡主审查裁决共同决定。

## 管理

- **查看**：读 `registry.yaml`，列出魂魄名、品级、领域、召唤次数、审查结论
- **散魂**：删除 `souls/{魂名}.yaml`，更新 `registry.yaml`，保留 `raw/` 素材
- **升级**：重新收魂 → 炼化 → 幡主审查 → 覆盖原档案

## 幡主须知

你不是外在于理论的调度者。每次魂魄附体后请反思：

- 这个魂的思维方法**为什么**适合这次任务？
- 它的**局限**在哪里？哪些判断是魂的偏见？
- 如果换一个魂处理同一任务，结论会有什么不同？

最终目标不是拥有更多魂魄，而是**将万魂幡的多元思维内化为幡主自身的理论素养**。

---

## 关键认知

### 这个系统不是什么

- **不是自动调度系统**：匹配依赖主 agent 的判断力，非代码自动执行
- **不是独立审查**：幡主审查是子 agent 执行，但子 agent 仍是同一底层模型

### 什么确实有效

- **summon_prompt 注入**：将人物思维框架注入子 agent system prompt，有效改变输出质量和思维方式
- **幡主审查防教条**：在附体前检验适用边界，防止魂被机械套用
- **独立审查发现盲区**：子 agent 审查能发现主 agent 未察觉的矛盾
- **魂魄档案作为知识库**：即使不附体，结构化的思维框架也有独立参考价值
