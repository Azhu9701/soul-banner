# 自动附体机制详解

## 执行流程

```
主agent收到任务
  → 判断需要 spawn sub-agent
  → 读取 registry.yaml
  → 遍历每个魂的 trigger，计算匹配度
  → 筛选匹配度 ≥ 70 的候选魂
  → 【幡主审查】判断匹配是否恰当
      → 候选魂的适用边界是否覆盖当前任务？
      → 是否存在教条式套用的风险？
      → 是否需要组合多个魂而非单一附体？
  → 幡主批准：
      → 是：读取该魂 souls/{name}.yaml
            → 将 summon_prompt 拼接至 sub-agent task 前缀
            → spawn 执行
      → 否：拒绝附体，正常 spawn sub-agent 或换魂
  → 无候选魂（最高匹配 < 70）→ 不附体
```

## 触发条件结构（trigger 字段）

```yaml
trigger:
  keywords: ["关键词1", "关键词2", ...]     # 直接关键词匹配
  domains: ["领域1", "领域2"]              # 领域匹配
  scenarios:                               # 场景描述（语义匹配）
    - "需要从本质分析复杂系统"
    - "做高风险决策评估"
  exclude: ["不触发的场景关键词"]            # 排除条件
```

**触发条件生成规则**：
- `keywords`：从素材中提取高频核心词 + 专业术语
- `domains`：直接复用 `domain` 字段
- `scenarios`：从素材中提炼该人物最擅长的 3-5 类场景
- `exclude`：明确不适用该魂的场景

## 匹配度评分公式

```
匹配度 = keywords命中数/总keywords × 30
       + domains命中数/总domains × 25
       + scenarios语义相似度 × 35
       + 品级加成(白+0, 绿+2, 蓝+5, 紫+8, 银+10, 金+12)
       - exclude命中 × 30
```

- **品级加成**：品级越高，同等匹配度下优先级越高
- **exclude惩罚**：触发排除条件直接扣 30 分

## 决策阈值

| 匹配度 | 行为 |
|--------|------|
| ≥ 70 | 候选进入幡主审查 |
| 50-69 | 不自动候选，主agent可选择手动指定 |
| < 50 | 不触发 |

若多个魂匹配度 ≥ 70，全部进入幡主审查。幡主从中选择最合适的魂（可以是最高分，也可以推翻分数选择更匹配的魂）。

## 手动指定

用户显式要求使用某个魂时（`用<人物名>来<任务>`），跳过自动匹配和幡主审查，直接注入指定魂魄。但主 agent 应提醒如果魂的适用边界不覆盖当前任务。

## 机制诚实说明

- **匹配公式是近似参考**：keywords 匹配为字符串重叠，scenarios 语义相似度无向量化实现。实际匹配依赖主 agent 对任务和魂的理解。
- **"自动附体"不是自动的**：依赖主 agent 记得执行流程。如果主 agent 跳过流程直接回答，魂不会附体。
- **幡主审查是最终判断**：公式计算的分值可由幡主推翻。
- **反馈闭环**：每次附体后须更新 summon_count 和召唤记录，积累数据校准匹配。

## 四种附体模式

### 模式一：单魂附体（Single Possession）

标准流程——匹配 → 幡主审查 → 单魂注入。适用常规任务。

### 模式二：多魂合议（Council）

**⚠️ 两阶段执行（关键）**：分析魂并行 → 等全部完成 → 幡主串行综合。幡主必须拿到真实输出，否则合议退化为"主agent模拟多视角"。

```
阶段一（并行）：N个魂同时独立分析
  → 魂A spawn + 魂B spawn + 魂C spawn
  → 等待全部完成，收集所有输出

阶段二（串行）：幡主辩证综合
  → 幡主子agent读取所有魂的实际输出
  → 找共识 → 找分歧 → 找盲区 → 辩证裁决
  → 输出合议报告
```

**适用条件**：任务跨领域、需多视角交叉验证、单一魂的 exclude 字段会遮掉关键视角。

**魂选择原则**：合议时选的魂应该**视角互补而非相近**。三个金魂合议 = 强化同一方向。乔布斯 + 费曼 + 列宁 = 审美 + 科学 + 阶级三层互补。

**幡主角色**：不是投票（"三个魂两个说该做就做"），而是辩证综合——各自在什么条件下对？矛盾能否统一在更高层次？

### 模式三：魂间辩论（Debate）

```
主agent识别对立维度 → 匹配两个立场相反的魂
  → 并行 spawn 魂A（正方）+ 魂B（反方）
  → 幡主读取双方论证
  → 输出：谁在什么条件下对？第三条路？
```

**魂配对原则**：选立场真正对立而非表面对立的魂。乔布斯（少即是多）vs Karpathy（验证驱动）有意义。乔布斯 vs 马斯克（都是极致主义者）无意义。

### 模式四：魂链接力（Chain）

```
魂A (阶段一) → 输出 → 魂B (阶段二) → 输出 → 魂C (阶段三) → ...
幡主在每个衔接点做质量审查和方向校验
```

**魂排序原则**：
- 先认识后行动：费曼（分析）→ 邓小平（判断）→ 罗永浩（叙事）
- 先拆解后建构：马斯克（成本重构）→ 乔布斯（体验设计）→ 列宁（方向审查）
- 幡主放在最后做终审

**衔接点审查**：幡主在每次交接时检查：前魂输出是否完整？后魂输入是否误解前魂？方向是否偏航？

## 反馈闭环

### 召唤记录格式

每次附体后在 `registry.yaml` 追加：

```yaml
召唤记录:
  - soul: "{魂名}"
    task: "{任务简述}"
    mode: "单魂/合议/辩论/接力"
    date: "YYYY-MM-DD"
    effectiveness: "有效/部分有效/无效"
    notes: "{判断依据}"
```

同步更新 `summoned_count`。

### 数据驱动校准

积累数据≥10条后：
- **匹配校准**：哪些魂的实际效果和匹配预期一致？哪些需要调整 trigger？
- **盲区发现**：哪些任务类型多次找不到合适魂 → 需收新魂
- **模式优化**：合议/辩论/接力——各模式效果如何？对什么任务类型最好？
- **过度依赖检测**：某魂被连续高频调用 → 是否需要补充该领域其他视角？
