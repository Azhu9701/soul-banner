# 自动附体机制详解

## 核心原则

**幡主也是被召唤的魂。** 主 agent 是纯协作者——不扮演任何魂，不做辩证综合，不做审查裁决。所有思考角色（包括幡主）都通过 spawn 注入 sumon_prompt 获得独立视角。

---

主 agent 角色边界详见 **[CLAUDE.md](CLAUDE.md#主-agent-角色定义)**：纯协作者——只调度、收集、存档。禁止扮演任何魂做实质性分析。

---

## 执行流程

```
主agent收到任务
  → 判断是否需要附体
  → 读取 registry.yaml
  → 遍历每个魂的 trigger，计算匹配度
  → 筛选匹配度 ≥ 70 的候选魂
  → 【幡主审查】: 统一 spawn 幡主子 agent 独立审查匹配
      → 审查适用边界、排除项、教条风险
      → 幡主返回：批准 / 否决（含理由）/ 附条件批准（调整建议）
      → 若否决或需调整，主 agent 重新匹配或修改注入方式
  → 批准 → 匹配的魂进入执行
  → 【执行阶段】:
      → 单魂：spawn 指定魂 → 输出
      → 合议：并行 spawn 分析魂 → 收集输出 → spawn 幡主辩证综合
      → 辩论：并行 spawn 对立方 → 收集输出 → spawn 幡主裁决
      → 接力：串行 spawn 魂链 → 每衔接点 spawn 幡主审查
  → 主 agent：更新 registry 召唤记录
  → 主 agent：存档 Obsidian
```

## 触发条件结构（trigger 字段）

```yaml
trigger:
  keywords: ["关键词1", "关键词2", ...]     # 直接关键词匹配
  domains: ["领域1", "领域2"]              # 领域匹配
  scenarios:                               # 场景描述（语义匹配）
    - "需要从本质分析复杂系统"
    - "做高风险决策评估"
  exclude: ["不触发的场景关键词"]            # 排除条件
```

**触发条件生成规则**：
- `keywords`：从素材中提取高频核心词 + 专业术语
- `domains`：直接复用 `domain` 字段
- `scenarios`：从素材中提炼该人物最擅长的 3-5 类场景
- `exclude`：明确不适用该魂的场景

## 匹配度评分公式

实际匹配由 `scripts/match.py` 执行（评分逻辑详见脚本源码）。以下为概念公式：

```
匹配度 = keywords命中权重 × 0.5
       + scenarios语义命中权重 × 0.3
       + domain重叠加分（≤0.2）
       + 认知距离调整（±0.05）
       × exclude惩罚（hard ×0.1 / soft ×0.5）
```

- **关键词权重**：高区分度词（3+字无停用词）×2.0，低区分度词 ×0.15-0.5
- **认知距离**：从 call-records.yaml 读取使用次数，`freshness = 1.0 / (1 + ln(usage + 1))`，归一化到 [-0.05, +0.05]。零召唤魂 +0.05，最高频魂 -0.05。**品级不影响匹配优先级**——品质由幡主审查保证，匹配由关键词/场景/领域/认知距离共同决定
- **exclude惩罚**：两级判定（hard ×0.1 / soft ×0.5），长词精确命中 = 硬排除，模糊/附属提及 = 软排除

## 对立度评分公式（辩论模式专用）

当候选魂 ≥ 2 时，对每对魂计算对立度。对立度 ≥ 60 的魂对自动推荐辩论模式。

```
对立度 = 方法论对立度(0-35)
       + 世界观立场对立度(0-35)
       + 领域边界互斥度(0-20)
       + 历史论战加成(0-10，确曾公开论战过的思想对手+10)
```

**方法论对立度**：魂A和魂B的核心方法论是否可以通约？
- 不可通约（如拉康精神分析 vs 第一性原理工程学）：25-35
- 部分可通约（如矛盾分析法 vs 阶级分析法）：10-24
- 高度可通约（如科学方法论 vs 第一性原理）：0-9

**世界观立场对立度**：两魂对同一问题的根本立场是否对立？
- 根本对立（如资本增殖逻辑 vs 劳动者发展逻辑）：25-35
- 视角差异但非对立（如渐进改革 vs 激进批判）：10-24
- 兼容互补：0-9

**领域边界互斥度**：是否一个魂的 exclude 字段恰好排除了另一个魂的核心领域？
- 互斥（如魂A exclude 阶级分析，魂B 核心领域=阶级分析）：15-20
- 部分重叠限制：5-14
- 无冲突：0-4

**推荐阈值**：

| 对立度 | 行为 |
|--------|------|
| ≥ 70 | 强烈推荐辩论，主动提示用户 |
| 60-69 | 推荐辩论，作为合议的替代选项 |
| 40-59 | 可辩论但不主动推荐 |
| < 40 | 不适合辩论（立场不够对立） |

**预计算的高对立度魂对**（基于已知方法论立场）：

| 魂A | 魂B | 对立度 | 对立轴 |
|-----|-----|--------|--------|
| 列宁 | 稻盛和夫 | 75 | 阶级分析 vs 经营哲学（阿米巴将劳资矛盾转化为劳动者内部竞争） |
| 鲁迅 | 乔布斯 | 80 | 文化批判 vs 消费美学（揭露瞒和骗 vs 体验至上） |
| 毛泽东 | Karpathy | 70 | 矛盾分析法 vs 四原则（政治动员 vs 工程极简） |
| 马斯克 | 波伏娃 | 75 | 第一性原理工程学 vs 存在主义女性主义（技术决定论 vs 社会建构） |
| 费曼 | 罗永浩 | 80 | 科学诚信（不欺骗自己） vs 品牌叙事（理想主义人设） |
| 邓小平 | 法农 | 75 | 渐进改革摸石头 vs 暴力去殖民化 |

此表非穷举——运行时根据对立度公式动态计算所有魂对。

## 决策阈值

| 匹配度 | 行为 |
|--------|------|
| ≥ 70 | 候选进入幡主审查 |
| 50-69 | 不自动候选，主agent可选择手动指定 |
| < 50 | 不触发 |

若多个魂匹配度 ≥ 70，全部进入幡主审查。幡主从中选择最合适的魂。

## 手动指定

用户显式要求使用某个魂时（`用<人物名>来<任务>`），跳过自动匹配计算，但仍须 spawn 幡主子 agent 审查。审查走简化版——只检查 exclude 是否触发、适用边界是否覆盖。若幡主否决，主 agent 告知用户风险并建议替代魂，但用户有最终决定权。

---

## 机制诚实说明

- **匹配公式是近似参考**：keywords 匹配为字符串重叠，scenarios 语义相似度无向量化实现。实际匹配依赖主 agent 对任务和魂的理解。
- **"自动附体"不是自动的**：依赖主 agent 记得执行流程。如果主 agent 跳过流程直接回答，魂不会附体。
- **幡主审查是最终判断**：公式计算的分值可由幡主子 agent 推翻。
- **反馈闭环**：每次附体后须追加顶层召唤记录，积累数据校准匹配。

---

## 分层数据访问（Token 优化）

匹配审查和附体执行使用不同数据层，禁止审查阶段读取完整魂 YAML。

| 阶段 | 数据源 | 读取内容 |
|------|--------|---------|
| 自动匹配 | registry.yaml | name / grade / trigger摘要 / gold_review |
| 幡主审查 | registry.yaml | 同上。若信息不足，仅读魂 YAML `_match_summary` 行 |
| 附体执行 | souls/{魂名}.yaml | 完整档案（含 summon_prompt） |
| 辩证综合 | — | 各魂的附体输出（原文，不做摘要） |

**禁止行为**：幡主在匹配审查时读取魂 YAML 的 summon_prompt 或 source_materials——这些字段仅附体执行时需要。

## 四种附体模式

### 模式一：单魂附体（Single Possession）

标准流程——匹配 → 幡主审查 → 单魂注入 → spawn 执行。适用常规任务。

### 模式二：多魂合议（Council）

**⚠️ 两阶段执行 + 幡主辩证综合（关键）**：分析魂并行 → 等全部完成 → spawn 辩证综合官串行综合。

```
阶段一（并行）：N个魂同时独立分析
  → spawn 魂A + spawn 魂B + spawn 魂C
  → 等待全部完成，收集所有输出
  → 主 agent 不做任何改动，纯收集

阶段二：spawn 辩证综合官子 agent 辩证综合
  → 注入辩证综合官 summon_prompt
  → 输入：所有分析魂的完整输出（原文，不摘要）
  → 辩证综合官独立输出：
      1. 共识（各魂一致的观点）
      2. 分歧（非对立而是视角互补？还是真正的矛盾？）
      3. 盲区（所有魂均未触及的系统性问题）
      4. 主要矛盾（当下最核心的对立统一）
      5. 行动纲领（可操作建议，区分各魂贡献和幡主补充）
  → 主 agent 接收辩证综合官输出
```

**辩证综合官 spawn 模板**：

```
你是{幡主魂名}之魂——万魂幡主。[summon_prompt全文]

## 当前任务

你收到了 N 个魂魄对同一问题的独立分析。请以你的辩证方法论，做辩证综合：

1. 找出所有魂的共识（他们都同意什么？）
2. 找出分歧（是视角互补还是真正的矛盾？）
3. 鉴定盲区（哪些关键维度所有魂都漏掉了？）
4. 锁定主要矛盾（当下最核心的对立统一是什么？）
5. 给出行动纲领（可操作建议，区分哪些来自哪个魂，哪些是幡主补充）

## 各魂分析

[魂A完整输出]
[魂B完整输出]
[魂C完整输出]
```

**适用条件**：任务跨领域、需多视角交叉验证、单一魂的 exclude 字段会遮掉关键视角。

**魂选择原则**：合议时选的魂应该**视角互补而非相近**。三个同方向金魂合议 = 强化同一方向。应选择覆盖不同维度的魂（如审美 + 科学 + 阶级分析三层互补）。

**辩证综合官角色**：不是投票（"三个魂两个说该做就做"），而是辩证综合——各自在什么条件下对？矛盾能否统一在更高层次？

### 模式三：魂间辩论（Debate）

**⚠️ 两阶段 + 幡主裁决**：

```
阶段一（并行）：spawn 魂A（正方）+ 魂B（反方）
  → 两魂从对立立场各自论证
  → 主 agent 收集双方输出

阶段二：spawn 幡主子 agent 裁决
  → 注入幡主 summon_prompt + 双方论证
  → 幡主独立输出：
      1. 谁在什么条件下对？
      2. 分歧的根源是什么（阶级立场？方法论差异？前提假设不同？）
      3. 是否存在第三条路（超越对立的更高统一）？
      4. 最终裁决及论证
  → 主 agent 接收幡主裁决
```

**魂配对原则**：选立场真正对立而非表面对立的魂。乔布斯（少即是多）vs Karpathy（验证驱动）有意义。乔布斯 vs 马斯克（都是极致主义者）无意义。

### 模式四：魂链接力（Chain）

**⚠️ 串行执行，衔接点审查为可选机制**。

接力模式的核心是迭代串行——前魂输出为后魂输入。衔接点审查**仅在触发以下条件时执行**：
1. 前魂输出有明显偏颇或遗漏关键维度
2. 后魂的 summon_prompt 可能导致误解前魂输出
3. 任务方向可能偏航（如分析结果与初始目标出现显著偏离）

不触发时不插入审查——直接衔接。触发时 spawn 幡主子 agent 审查衔接点：
- 魂A输出是否完整？是否遗漏关键信息？
- 魂B的输入是否误解了魂A？
- 方向是否偏航？
- 批准 → 进入阶段二 / 否决 → 要求魂A修正

首测案例（厂二代分析，2026-04-30）：四魂均为金魂，输出质量高，逻辑链自然衔接（阶级位置→矛盾定性→诚实检测→行动方案），未触发衔接审查。各魂主动回应前魂输出（毛补充列三点/鲁五刀按列三类型定制/邓三步走直接回应鲁诚实标准），证明高质量链中衔接审查是冗余。

**魂排序原则**：
- 先认识后行动：费曼（分析）→ 邓小平（判断）→ 罗永浩（叙事）
- 先拆解后建构：技术分析（成本重构）→ 体验设计（产品化）→ 幡主（方向审查）
- 幡主放在最后做终审

### 模式五：幡主学习（Banner-Master Study）

**目标不是产出答案，而是训练幡主的辩证素养。** 两个方法论最不可通约的魂互相解释对方的框架，幡主在冲突中学习。

```
阶段一（并行）：魂A向魂B学习 + 魂B向魂A学习
  → spawn 魂A：解释魂B框架的核心假设、成立条件、以及魂B揭示的自身盲区
  → spawn 魂B：解释魂A框架的核心假设、成立条件、以及魂A揭示的自身盲区
  → 主 agent 收集双方输出

阶段二（并行）：互审对方的解释
  → spawn 魂A：魂B对我的解释哪里对了？哪里是误读？为什么会有这种误读？
  → spawn 魂B：魂A对我的解释哪里对了？哪里是误读？为什么会有这种误读？
  → 主 agent 收集互审结果

阶段三：spawn 幡主子 agent 学习总结
  → 输入：双方的解释 + 双方的互审
  → 幡主独立输出：
      1. 学到了什么（之前没意识到的视角/假设/盲区）
      2. 两个框架在什么条件下各自成立（不是"谁对谁错"而是"谁在什么条件下对"）
      3. 哪些自身假设被这次学习动摇了
      4. 新产生的问题（而非答案）
  → 主 agent 接收幡主学习总结

阶段四（学习答辩）：spawn 魂A + 魂B 审核幡主学习总结
  → 魂A 必须回答：
      1. 幡主对我的框架的理解，哪里对了？
      2. 哪里是误读？这种误读的根源是什么（时代间距？立场差异？方法论不可通约？）？
      3. 如果我是幡主的老师，我会给他什么反馈以纠正理解？
  → 魂B 同样回答
  → 判定规则：
      · 两魂一致指出的同一系统性误读 → 幡主学习未通过，需重新进入阶段一
      · 仅一魂指出的问题 → 幡主选择性修正学习总结，追加「答辩修正」段落
      · 两魂均未指出系统性误读 → 答辩通过
  → 答辩结果写入学习记录，替代自评作为 effectiveness 评级依据
  → 季度第三方金魂（非参与魂）抽查历史学习记录，判断是否存在系统性自欺模式
```

**学习评价标准**（由答辩结果决定，替代自评）：
- 通过：幡主学习总结被两魂确认无系统性误读
- 条件通过：存在个别误读但已修正
- 未通过：存在至少一个两魂一致指出的系统性误读，需重新学习

**魂选择原则**：选对立度最高的魂对（参见对立度评分公式）。对立度 ≥ 70 的魂对最适合此模式。**禁止选相近魂**——两个方向一致的金魂合议是强化，不是学习。学习的价值来自不可通约的张力。

**与辩论模式的区别**：
| 维度 | 辩论 | 幡主学习 |
|------|------|---------|
| 目标 | 裁决——谁在什么条件下对 | 成长——幡主学到了什么 |
| 产出 | 行动纲领/裁决 | 新问题/被修正的假设 |
| 魂的关系 | 对抗——各自捍卫立场 | 互读——各自尝试理解对方 |
| 幡主角色 | 裁判——站外面裁决 | 学生——站里面学习 |

**使用场景**：
- 幡主发现自己对某魂过度依赖（连续调用 > 3 次）→ 用该魂和其最高对立魂做学习
- 幡主审查时发现某个魂的盲区反复出现 → 对该魂和揭示其盲区的魂做学习
- 幡主想测试自己是否真正理解了某个魂的排除边界 → 用该魂和其排除域魂做学习
- 定期（每 30 天或每 10 次附体后）→ 选两个高对立度魂做学习，防止幡主思维同质化

**记录格式**：召唤记录中 `mode: 幡主学习`，`effectiveness` 由阶段四答辩结果决定：
- `有效` — 答辩通过（无系统性误读）
- `部分有效` — 条件通过（有个别误读已修正）
- `无效` — 未通过（系统性误读，需重新学习）

**学习成果归档**：每次幡主学习完成后，成果按五类归档至 `reviews/幡主学习/{YYYY-MM-DD}-{魂A}vs{魂B}/`：
1. `学习总结.md` — 幡主学习总结 + 答辩修正
2. `魂A答辩.md` — 魂A的答辩输出
3. `魂B答辩.md` — 魂B的答辩输出
4. `结构化归档.yaml` — 五类分类（trigger_calibration/blindspot_mapping/boundary_refinement/pairing_insight/self_revision）
5. `补丁提案.yaml` — 前三类自动生成的补丁提案草稿（如有）

## 魂修正案流程（Soul Amendment Process）

**目的**：为修改魂的 summon_prompt / trigger / exclude / 适用边界建立合法的宪法程序。解决「审查可以批判魂，但批判不改变魂的行为」的场域论断裂。

### 权力分配

| 角色 | 权力 | 边界 |
|------|------|------|
| 幡主 | **提案权**——基于学习成果或审查发现提出修正案 | 不能单方面修改；提案必须引用具体证据 |
| 被修改魂 | **申辩权**——spawn 回应修正案是否符合自身思想内核 | 不能否决提案，只能提供申辩意见 |
| 审查委员会 | **裁决权**——轮值审查官裁定批准/否决/附条件批准 | 不能主动提案；裁决必须附论证 |

### 流程

```
1. 提案
   幡主撰写修正提案，包含：
     - 修改目标（summon_prompt 哪一段？trigger 哪个字段？边界哪个方向？）
     - 修改理由（引用审查报告/幡主学习成果/附体失效证据）
     - 修改后文本
     - 预期效果（修改后附体应产生什么变化？）

2. 申辩
   spawn 被修改魂子 agent，提交修正提案
   魂必须回答：
     - 这个修改是否符合我的思想内核？为什么符合/不符合？
     - 修改后，我的方法论在什么条件下会失效？
     - 有没有更好的方式达到提案想达到的效果？
   魂的申辩是咨询性的——不具否决权，但审查委员会必须考虑

3. 裁决
   轮值审查官 spawn 审查提案 + 魂的申辩
   审查官裁定：
     - 批准：修改写入魂 YAML / registry
     - 否决：说明理由，提案退回幡主
     - 附条件批准：修改生效但附带监测条件（如 5 次附体强制评分 + 审查官复核）
   裁决必须包含论证——不能只说「批准」或「否决」

4. 版本标记
   批准后，修改内容写入对应字段
   patches[] 追加新补丁记录，标记：
     - version: N+1
     - applied_at: YYYY-MM-DD
     - source: "魂修正案-{提案编号}"
     - approved_by: "{审查官}-{裁决日期}"
   修改前版本保留在版本历史中（不删除）

5. 生效监测
   补丁生效后 5 次附体强制执行有效性评分 + 审查官复核
   与修改前 5 次对比：
     - 有效性提升 → 补丁固化
     - 无变化 → 补丁维持，标记为「中性」
     - 有效性下降 → 触发回滚，恢复修改前版本
```

### 不可修改项

以下字段**不可通过修正案修改**，仅可通过重新收魂→炼化→审查修改：
- `name`（魂的标识）
- `grade`（品级，仅审查可裁定）
- `source_materials`（素材来源，历史锚点）

### 紧急回滚

若补丁生效后发现严重问题（如魂的输出方向性错误连续2次），幡主可临时冻结补丁（不删除），同时发起紧急复审。审查委员会48小时内裁定永久回滚还是修复后重新生效。

## 学习成果结构化归档

**目的**：将幡主学习成果从 call-records.yaml 的一行 notes 升级为结构化、可检索、可触发补丁提案的知识资产。

### 五类归档

| 分类 | 说明 | 触发操作 |
|------|------|---------|
| `trigger_calibration` | 发现某魂的 trigger 应调整 | 自动生成 trigger_adjustment 补丁提案 |
| `blindspot_mapping` | 发现某魂的系统性盲区 | 自动生成 blindspot_awareness 补丁提案 |
| `boundary_refinement` | 发现某魂的适用边界需调整 | 自动生成 boundary_refinement 补丁提案 |
| `pairing_insight` | 发现魂间搭配效果 | 进入魂配对推荐知识库 |
| `self_revision` | 幡主自身偏见/假设修正 | 进入幡主个人成长追踪（不做系统级操作） |

### 归档路径

```
reviews/幡主学习/{YYYY-MM-DD}-{魂A}vs{魂B}/
├── 01-魂A解释魂B.md
├── 02-魂B解释魂A.md
├── 03-魂A互审.md
├── 04-魂B互审.md
├── 05-幡主学习总结.md
├── 06-魂A答辩.md
├── 07-魂B答辩.md
├── 结构化归档.yaml
└── 补丁提案.yaml  # 前三类有内容时生成
```

### 补丁提案自动生成规则

```
if 归档分类 in [trigger_calibration, blindspot_mapping, boundary_refinement]:
    生成 补丁提案.yaml:
      - 提案来源（哪次学习/审查）
      - 目标魂
      - 补丁类型
      - 补丁内容草稿
      - 引用证据（学习总结中的具体段落）
      - 状态：待幡主审核
```

幡主审核补丁提案草稿后，可发起魂修正案流程。

## 散魂仪式

**散魂不是删文件，是知识生产。** 当一个魂触发三维遗忘审查（使用价值维/可替代性维/方法论前提维）且审查委员会裁定散魂时，执行以下仪式。

### 触发条件

散魂由三维遗忘审查触发，不是幡主随意决定：

| 维度 | 触发条件 | 散魂建议 |
|------|---------|---------|
| 使用价值维 | 零召唤 ≥ 90 天 且 非稀缺领域 | 强烈建议散魂 |
| 使用价值维 | 连续 5 次无「有效」且复审确认 | 建议散魂 |
| 可替代性维 | 另一魂覆盖重叠领域且表现显著优于本魂 | 建议散魂 |
| 方法论前提维 | 核心前提已被历史证伪且方向判断能力已失效 | 建议降品，严重者散魂 |

### 仪式步骤

```
1. 终末审查
   由当前轮值审查官 spawn 执行
   使用与入幡审查相同的板块结构（审查类型+板块0-7）
   审查类型：终末审查（区别于入幡审查的品级判定）
   必须回答四个问题：
     Q1: 该魂没有被使用是因为——
         (a) 幡主的任务分布不覆盖其领域
         (b) 其方法论前提已消失
         (c) 有更好的魂替代了它
         (d) 品级本身是错的
     Q2: 如果该魂在今天才被收魂炼化，品级会不同吗？
         （测试品级标准是否随时间自修正）
     Q3: 该魂在幡期间是否产生了间接价值？
         （审查对照角色/学习答辩对立面/匹配审查参照物）
     Q4: 散魂后，该魂覆盖的领域是否出现空白？
         如果是——谁来补？是否需要收新魂？

2. 魂的申辩
   spawn 被散魂子 agent，提交终末审查四问
   魂必须回答四问（这是魂在幡中的最后一次附体）
   魂的申辩成为终末审查报告的附件

3. 知识归档（非删除）
   - 魂 YAML 移动至 raw/已散魂/{魂名}.yaml
   - 终末审查报告保存至 reviews/散魂/{魂名}-终末审查-{日期}.md
   - registry.yaml 中标记 status: scattered（不删除条目）
   - 保留 scattered_at 和 scatter_reason 字段
   - raw/ 素材完整保留

4. 幡主学习
   散魂审查报告的四个问题就是学习材料
   幡主 spawn 反思：
     - 这个魂为什么进入幡？当时收魂的判断是什么？
     - 为什么后来没有被使用？是幡主的问题（任务分布）还是魂的问题（品级/适用性）？
     - 从这个魂的生命周期中学到了什么关于收魂标准/品级判定/匹配机制的教训？

5. 品级标准反馈
   若被散魂是金魂 → 触发金魂三条标准复审
   累计散魂的金魂数 ≥ 3 时 → 重新审议金魂标准
```

### 裁决后自动更新（强制执行）

终末审查裁决做出后，主 agent 必须**立即自动**执行以下更新，无需等待用户确认：

**若裁定散魂**：
- 执行步骤 3（知识归档）：移动 YAML → 标记 registry → 保存终末报告
- 更新 committee/state.json pending_tasks（标记对应散魂任务为已完成）

**若裁定维持（不散魂）**：
- 更新 registry.yaml 中该魂的 gold_review（追加终末审查结论）
- 更新 registry.yaml 中该魂的审查记录（追加终末审查条目）
- 更新 committee/state.json pending_tasks（标记对应终末审查任务为已完成）
- 若有观察期条件，在 committee/state.json verification_conditions 中追加到期检查项

**原则**：用户不应需要手动说「更新」。裁决即生效，生效即落盘。

### 散魂后领域空白处理

若终末审查 Q4 判定领域出现空白，启动**定向收魂**——不是为了收藏，而是为了填补已确认的系统性缺口。定向收魂的 trigger 条件比常规收魂更精确：基于散魂分析中确认的缺失维度来设计搜索关键词。

## 审查报告强制板块

审查报告强制板块顺序（不可变，详见 SKILL.md 审查章节）：

```
[PRE] 审查类型判定（封闭档案审查/开放实践审查）
[0] 审查者的前提假设（3个问题）
[1] 历史结构条件分析（4个强制问题）
[2] 肯定方面（与唯物主义的相通之处）
[3] 批判方面（阶级分析、适用边界、前提假设）
[4] 金魂第(3)条条件化判定（3a/3b/3c，如适用）
[5] 品级裁定（升/降/维持，有论证）
[6] 适用边界（可用/禁用领域）
[7] 标准应用一致性论证（与其他案例比较）
```

### 审查类型判定（板块 PRE）

在进入所有其他板块之前，审查者必须先判定审查类型。判定标准：被审查者是否仍在世并在实践中？

- **封闭档案审查**：已去世。思想遗产是封闭档案——不会产生新事实。金魂第(3)条评估生前在历史档案中可查证的自我修正实践。
- **开放实践审查**：仍在世。思想和方法论仍在演化。金魂第(3)条使用3a/3b/3c条件化判定。审查结论是阶段性的——必须标注「基于截至YYYY-MM-DD的已知事实」。

判定结果格式：
```
- 类型：封闭档案审查 / 开放实践审查
- 判定依据：[一行说明]
- 对第(3)条的影响：[用哪套规则]
```

### 审查者前提假设（板块 0）

审查者在给出任何判断之前，必须回答三个问题：

1. **审查标准的自身前提**：本审查所依据的标准（金魂三条、品级体系、阶级分析方法）自身的历史前提和阶级前提是什么？在被审查者所处的历史条件下，该标准的哪些部分可能不适用？

2. **结构性不对等造成的盲区**：审查者与被审查者之间存在什么结构性不对等？（时代间距、社会位置、信息不对称、权力关系）这种不对等可能使审查者无法看到被审查者的什么？

3. **换位检验**：如果换一个与被审查者立场/时代/处境更接近的审查者来执行本审查，哪些判断可能不同？给出至少一条具体的可能差异。

### 历史结构条件分析（板块 1）

回答四个强制问题：

1. **时代约束**：被审查者所处历史时期的结构性限制是什么？哪些条件在当时的客观环境中不可用？

2. **空间约束**：在上述约束下，被审查者实际可用的手段有哪些？列出具体边界。

3. **尝试了什么**：被审查者实际尝试了哪些制度化/组织化/自我修正的途径？哪些成功？哪些失败？

4. **失败原因归属**（二选一，必须给出判断）：
   - **结构不可为**：客观结构限制是失败的主要原因
   - **个人可做而未做**：存在可操作但未采取的路径，个人选择是主要原因

### 反向审查问卷（板块后置）

审查者完成审查报告后，提交以下三个问题给被审查者（魂），以 spawn 方式独立回答。被审查者的回答成为审查报告的正式附件。

**问题一**：审查者对我的前提假设是什么？其中哪一条在我所处的历史条件下不成立？

**问题二**：审查者用来评判我的标准，其自身适用的条件边界是什么？若审查者自己处于我的历史位置，该标准的哪些要求他也做不到？

**问题三**：如果由我来审查审查者，我会指出其审查方法中的什么盲区？给我一条具体的、审查者自己没有意识到的批评。

执行流程：审查者完成报告 → spawn 被审查者子 agent 注入三个问题+审查报告 → 被审查者独立回答 → 回答追加为「反向审查附件」→ 审查者可补充注释（不修改原结论）
